## 临界观察复核清单
只展示仍可申购、评分接近建议阈值的 `可选观察` 股票。该清单不是默认申购指令，也不自动占用乙组资金；它用于在券商融资截止前补充孖展倍数/金额、额度紧张度、利率手续费、券商融资截止、招股书估值/基石和舆情分歧。只有至少两个融资截止前需求/额度类强热度信号且成本可接受时，才考虑升级到现金/甲组；乙组仍需单独核价。低利率是成本条件，不算作热度信号。若回测稳定性审计显示漏掉强收益集中在临界观察升级，应先运行 `prepare_borderline_upgrade_template.py --priority-levels P0` 生成主年份高临界分补采清单，并用 `normalize_conflict_research_input.py` 校验填回资料；P0 可复核或明确缺数据后再扩展 P1。

## 招股书深挖优先队列
只展示仍可申购且可能改变推荐或资金动作的股票。`建议申购` 为 P0，临界 `可选观察` 为 P1。必须列出 HKEX 招股书链接或文件缺口，并说明要查的重点：估值/发行市值、收入和利润质量、基石/禁售、募资用途、主要风险、-B/-P 管线或商业化情况。不要对所有股票机械深挖。

## 融资核价清单
集中展示仍可申购且可能需要甲组/乙组或现金多手核价的股票。必须列出当前融资状态、预案金额、孖展倍数/金额、额度紧张度、利率和手续费、券商融资截止、占款天数、资金窗口冲突，以及是否仍只是 `乙组待闸门`。未补齐这些数据时，不要把乙组候选写成已执行指令。

## 建议申购执行风险复核
当回测稳定性审计显示 `建议申购` 假阳性集中在一手期望、资金效率、硬科技估值或需求验证时，应先运行 `prepare_execution_risk_template.py --priority-levels P0`。P0 优先处理高分乙组和分数段融资效率背离样本；补采字段应包括融资利率、手续费、计息天数、情景首日涨幅、情景配售率/可信上限、孖展热度、额度状态、估值/发行市值、同业可比、基石/禁售和硬科技商业化验证。填回后必须先用 `normalize_conflict_research_input.py --markdown` 校验填回质量：`待填回` 代表模板尚未填入用户证据，`时间无效` 才代表已有内容但不能证明早于融资截止。之后再用 `audit_financing_efficiency.py --scenario-json` 做逐股资金效率审计。不得用最终一手中签率、配售结果、暗盘或首日表现作为锁单输入。只有 P0 已可复核或明确是数据缺口后，才扩展到 P1。
闸门信号必须按独立类别展示：孖展倍数和孖展金额同属 `孖展规模`，不能重复算作两个强热度信号；还需多券商一致、额度紧张/截止提前、尾日需求加速等其它独立信号。

## 融资锁单时间表
按招股截止日倒排 T-2/T-1/T-0 动作，只展示仍可申购且可能影响现金/甲组/乙组选择的股票。必须列出阶段、当前状态、必须马上做的核价或热度动作、以及禁止拖延到配售结果/暗盘/首日后才决定融资的提示。
T-0 样本只能使用已经确认的券商额度、利率、费用和截止时间；若乙组候选仍缺至少两个融资截止前需求/额度类强热度信号或成本不可接受，应退回甲组/现金或跳过。

## 默认资金排期建议
| 优先级 | 股票 | 推荐 | 资金动作 | 预计占用现金 | 锁定窗口 | 排期结论 |
|---:|---|---|---|---:|---|---|
只展示仍需占用默认现金的新申购安排。已截止、待暗盘或只复盘的股票不要出现在这里。
`乙组候选` 但融资热度闸门未满足时，资金动作应写作“现金/甲组预案；乙组待闸门”，预计占用现金只按预案计算；闸门满足后才按乙组可执行占款。

## 同窗口取舍复核
只展示仍可申购且默认 HKD 550,000 现金无法同时覆盖的重叠窗口股票。必须列出待取舍股票、当前已排入的冲突对象、锁定窗口、预计占用现金，以及融资截止前必须完成的复核动作。
复核动作必须使用事前可见信息：招股书深挖、T-1/T-0 孖展热度、额度紧张度、融资利率/手续费、事前占款效率（入场费、拟申购金额、融资息费打平幅度）和舆情分歧。不能用一手中签率、配售结果、暗盘或首日涨跌来决定是否替换默认排期。
若冲突票的事前证据明显强于已排入票，可以在券商融资截止前替换；若复核未完成，维持默认排期，不追加占用同一笔现金。

## 资金锁定检查
| 股票 | 锁定窗口 | 预计占用现金 | 结论 |
|---|---|---:|---|
已截止且用户未记录实际参与的股票，预计占用现金应为 0，并转入上市表现复盘；不要让它们挤占仍可申购股票的现金窗口。

## 上市表现复盘
仅在暗盘、配售结果或首日表现可得时展示。
当 `build_recommendation_report.py` 提供 `--review-json` 时，必须按股票代码合并年度已上市数据，展示首日涨跌幅、公开超购、一手中签率和一手期望毛利，并说明它们只用于复盘，不得改变事前推荐评分。
复盘行必须给出下一次动作：最终弱热度导致的乙组/现金降级、最终强热度导致的 T-1/T-0 升级复核、以及一手期望过低时的资金效率提醒。
如果用户提供实际申购结果，应先归一化为股票、申购金额/手数、中签股数/手数、招股价、卖出价或暗盘/首日涨跌幅、融资利息和手续费，再展示实际交易毛利、扣成本收益、申购金额收益率和现金收益率。输入缺失时只列缺口，不要臆测。

## 年度回测
仅在用户要求回测或优化策略时展示。必须区分事前推荐模型、申购截止前融资预决策、上市后复盘模型，避免未来数据泄露。除首日涨幅外，应展示一手期望毛利等资金效率指标，避免只优化涨幅。
应展示 `资金窗口压力测试`：用默认 HKD 550,000 现金、同一锁定窗口不可重复使用、按默认事前排期规则，模拟历史 `建议申购` 股票能排入多少只、因重叠窗口跳过多少只、峰值现金占用、排入和跳过的一手期望毛利。必须同时展示一手期望合计、平均一手期望和一手期望覆盖样本数，避免样本数不同或数据缺失造成误读。该表只评估资金排期约束和机会成本，不代表实际获配收益。若已经使用 `事前效用组合最优` 后冲突跳过组合在平均一手期望、合计一手期望、强收益率或其他关键复盘指标上仍强于排入组合，必须把它归因为残余数据缺口，运行 `prepare_conflict_research_template.py --priority-levels P0` 先生成排期边界补采清单；不要继续机械拟合排序代理。
应展示 `排期排序敏感性`：比较基础分数优先、分数+低入场费优先、融资分层+分数+低入场费、低入场费+分数优先等事前排序。排序只能用申购截止前可见字段；首日表现和一手期望毛利只能作为复盘检验，不得参与当时排期。敏感性判断应优先比较平均一手期望，再用合计一手期望识别机会成本。若某个事前 tie-breaker 稳定改善排入组合并降低冲突跳过组合机会成本，可作为默认排期 tie-breaker，但不得证明低入场费本身足以申购或融资。
应展示 `历史孖展覆盖审查`：报告乙组候选数量、历史孖展覆盖数量、缺数据数量、覆盖率、严格闸门满足/不满足数量和缺数据候选。没有历史孖展 JSON 时，应明确乙组执行效果不能验证，并给出 `prepare_margin_history_template.py` 的补采命令。覆盖率低于 70% 时只能作为数据缺口审查，不得证明乙组执行策略有效。
历史孖展覆盖率只计算时间有效记录；`timing_valid_row_count=0`、`timing_confidence=未确认` 或记录时间晚于券商融资截止的样本必须单列为时间无效，不得归入已覆盖但闸门未满足。
应单独展示 `临界观察队列复盘`：事前为 `可选观察` 且接近建议阈值的股票，报告其正收益率、强收益率、平均一手期望，以及最终强/弱热度代理。该结果只用于证明是否值得 T-1/T-0 升级复核，不得自动变成事前推荐规则。
若该队列贡献了主要假阴性，下一步应生成临界观察升级补采模板，而不是直接降低 `建议申购` 分数阈值。填回资料必须通过融资截止前时间校验和未来数据污染校验后，才可作为下一轮融资热度或升级复核输入。
若 `建议申购` 贡献了主要假阳性，下一步应生成建议申购执行风险补采模板，重点复核逐股融资成本、情景配售率、打平涨幅、估值和需求验证，而不是简单降低硬科技/高分行业权重。
应展示 `评分分层校准`：按事前分数段展示正收益率、强收益率、平均首日和一手期望，并说明这只是阈值敏感性检查。若高分段没有稳定优于次高分段，必须明确不要机械提高阈值，而应强化招股书深挖、融资热度和成本闸门。
生成稳定性审计 JSON 后，应运行 `plan_backtest_next_actions.py` 输出 P0/P1/P2 下一步动作计划、迭代闸门和 `证据闭环`。该计划应列出具体命令和防泄露约束：优先补 P0 乙组孖展、建议申购执行风险、临界观察升级和同窗口残余冲突；当闸门显示 `先补 P0 证据` 或 `回退策略改动` 时，不得继续机械调阈值。若多个 P0 工作流同时存在，应先运行 `prepare_p0_evidence_pack.py --backtest-json year.json --stability-json stability.json` 汇总 P0 股票数、跨领域重叠、合并补采工作表和生成/归一化/复核命令。若跨领域重叠较多，应先用 `--consolidated-csv` 生成按股票去重的补采工作表，减少重复录入；填好后用 `split_p0_consolidated_input.py --domain all --output-dir dir` 拆回四个 P0 域 CSV。最后运行 `audit_expert_readiness.py --backtest-json year.json --stability-json stability.json --report year.md --primary-year YYYY --json > expert-readiness-YYYY.json`，并用 `--p0-readiness-json domain=path` 接入已归一化 JSON 或拆分 CSV；专家闸门会自动用对应 normalizer 做时间和证据污染校验，并输出 `P0 下一批补证据清单`。该清单按待闭环领域数、跨领域数、事前分数排序，应用作下一轮补采顺序：先补跨领域高分且仍待填回的股票，再处理单领域或低分样本。若需要公开检索辅助，先运行 `prepare_p0_research_queries.py --input expert-readiness-YYYY.json` 查看按股最小闭环清单，再运行 `--csv > p0-research-queries-YYYY.csv` 生成逐股可填证据台账；查询语句只用于寻找申购截止前的券商孖展、HKEX 招股书、情景假设和辅助舆情证据，不能把配售后、暗盘或上市后信息填回 P0。P0 台账不能因单条孖展或单条舆情记录就判为可复核：若已生成孖展任务，至少要有一条时间有效的热度/成本/计息天数记录；若已生成情景任务，要有申购前首日涨幅和配售率/可信上限假设；若已生成招股书任务，要有招股书链接或来源及摘要。额外券商行和舆情行用于交叉验证，但空白时不应覆盖核心证据缺口。若公开申购前来源确实不可得，必须填 `search_attempted_at`、`search_source`、`unavailable_reason` 和 `search_note`；归一化后只有无污染的 `已尝试缺口` 可被显式接受。填好台账后优先运行 `run_p0_evidence_pipeline.py --consolidated p0-consolidated-YYYY.csv --ledger p0-research-queries-YYYY.csv --output-dir p0-evidence-run-YYYY --backtest-json year.json --stability-json stability.json --report year.md --primary-year YYYY`；流水线会归一化台账、只合并时间有效且未污染的行或无污染的已尝试缺口、拆回分域 CSV，并可复跑专家闸门。该合并默认只填空字段，不覆盖人工已确认内容。只有专家闸门显示无 error、无阻塞 warning、无 P0 待补证据，才可说当前没有自动化优化方向。`--accept-p0-data-gaps` 只接受 `已尝试缺口`，不能用于普通 `缺数据`、`时间无效`、`证据污染` 或空白模板。
当用 2026 回测验证当前推荐策略时，应先展示或引用 `策略一致性审查`：比较当前报告桶位和 2026 事前回测桶位，只把仍可申购样本的不一致计为策略漂移；已截止、暗盘或已上市样本的差异属于报告上下文处理，不应削弱 2026 单年主结论。
若当前报告合并了招股书深挖 JSON，审查应把由深挖造成的降级或升级标记为 `深挖覆盖差异`，不直接计入基础策略漂移；但仍需人工复核深挖摘要是否足够可靠。

## 多年份回测
优化策略时作为当前年后的压力测试展示。应给出主评估年份、年份权重、年度表现、近因加权表现、融资分层近因加权、跨周期融资压力审查、一手期望毛利和专家审查结论。
当用户强调当前市场或 2026 年时，报告必须说明当前年是主证据，旧年份只用于识别跨周期风险和融资纪律问题；主结论以 2026/当前年单年回测为准，旧年份不得作为直接调参样本稀释或推翻 2026 结论。若用户明确说过去环境可能不同、主要看 2026，则不要自动生成多年份调参结论；除非用户要求压力测试，否则只输出 2026 单年策略评估和待补证据。
专家审查结论必须先比较主评估年份的当前策略 vs 原策略，再展示近因加权旁证。比较项至少包括平均首日表现和一手期望毛利；若两者方向相反，必须提示不要只按涨幅优化，应优先复查资金效率、回拨概率和融资成本。
必须区分 `近因权重` 和 `有效权重`。详情页或行业覆盖不足的年份应设为 0 有效权重，只作为覆盖风险提示，不应进入加权表现。
必须展示 `跨周期融资压力审查`：列出旧年份乙组候选数量、强收益率和一手期望；若旧年份乙组表现弱于当前年或一手期望偏低，必须明确乙组候选跨周期不稳定，不能默认执行乙组，只能等待融资截止前热度、成本和资金窗口同时通过。
策略调参时应说明是否复用了同一批 JSON 快照；如果每轮都重新联网抓取，应提示数据覆盖变化可能影响分组。

## 融资闸门回测
当有历史融资热度数据时展示。必须报告乙组候选样本、热度覆盖率、覆盖率审查、闸门满足/不满足/缺数据三组表现，并声明只使用融资截止前数据。若乙组候选热度覆盖率低于 70%，不得把该回测解读为乙组执行策略有效，只能作为数据采集缺口审查。
覆盖率不足时，应运行 `prepare_margin_history_template.py` 生成乙组候选历史孖展补采模板，优先补 broker、observed_at、source_published_at、preclose_confirmed、broker_cutoff_at、margin_multiple、margin_amount_hkd、quota_status、financing_rate_pct、cutoff_note、source 和 excerpt。若乙组候选很多，先用 `--priority-levels P0`；P0 必须覆盖主年份高分乙组候选里缺申购前孖展热度或严格闸门未满足的样本，即使入场费不高。优先级只能依据事前分数、入场费、已有热度覆盖状态和时间有效性，不得依据首日、最终超购或一手中签率。填回后先运行 `normalize_margin_history.py --markdown` 检查 `待填回`、`时间无效` 和 `证据污染`，再输出 JSON 给 `backtest_margin_gate.py`。
同窗口残余冲突仍高时，应运行 `prepare_conflict_research_template.py --priority-levels P0` 生成排期边界补采模板，优先补 residual group 内“默认排入”和“边际跳过/近效用挑战者”的 observed_at、source_published_at、broker_cutoff_at、margin_multiple、margin_amount_hkd、quota_status、financing_rate_pct、fees、loan days、scenario allotment-rate range、HKEX 招股书链接、估值/基石/禁售深挖摘要和来源摘录。填回模板后必须先运行 `normalize_conflict_research_input.py --markdown` 查看填回质量，再输出 JSON；只有通过时间校验且没有配售后/上市后证据污染的 `items_by_stock` 可以作为 `--margin-heat-json` 输入继续回测。P0 可复核或明确缺数据后才扩展到 P1。
如果历史热度来自 CSV/JSONL 批量导入，必须说明未确认 `preclose_confirmed` 的样本不能作为乙组执行证据。
`observed_at`、`source_published_at` 和 `broker_cutoff_at` 应尽量精确到分钟；`observed_at` 必须能解析且不得晚于券商融资截止或公开招股截止。若提供了 `source_published_at`，它同样不得晚于券商融资截止；否则只能说明事后找到过申购前数据，不能证明当时可用于融资决策。若券商截止时间有具体时刻但记录/来源发布时间只有日期，不能确认早于截止，只能作为复盘资料。若记录时间或来源发布时间晚于截止，即使 `preclose_confirmed=是` 或使用 `--assume-preclose`，也不能进入乙组执行闸门。

## Style Rules

- Stock names should be Chinese. If not verified, write `代码XXXXX.HK（中文名待核实）`.
- Do not include source status words in stock names. Render `今日暗盘`, `今日上市`, `跌穿上市价`, or similar phrases in status/review text instead.
